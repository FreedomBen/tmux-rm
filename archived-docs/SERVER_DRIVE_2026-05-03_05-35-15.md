## Findings

### F-01 — `/` redirects new user to a bare "Not Found" page

- **Severity:** major
- **Repro:**
  1. Boot a fresh container with no admin user.
  2. Open `http://<host>:<port>/` in a browser.
- **Expected:** Either landing on the setup form, or a page that
  explains how to obtain the setup token (the URL with the token is
  printed to the container log, but a non-technical user opening the
  app in a browser has no obvious way to discover it).
- **Actual:** The browser is server-redirected to `/setup` (no
  query string), which the router treats as "Not Found" and renders
  a blank page with the literal text `Not Found` in the top-left
  corner. No link, no copy, no styling.
- **Screenshot:** `drive-artifacts/01-setup-no-token.png`
- **Note:** With the token present, the setup page is *also*
  inaccessible from a host browser (see F-02). The empty-handed
  flow being broken is a separate UX issue from the loopback gate.

### F-02 — Container deployment cannot complete setup from a host browser

- **Severity:** blocker
- **Repro:**
  1. Run termigate inside a container (`make run-container`).
  2. From a browser on the same host machine, open the URL the
     container printed: `http://127.0.0.1:<port>/setup?token=<t>`.
- **Expected:** The first-run setup form renders so the host owner
  can create the admin account.
- **Actual:** Server returns `404 Not Found`. The container log
  shows `/setup access denied: non-loopback peer, ip=192.168.2.2`
  — Podman's userland proxy NATs the source IP to the bridge
  gateway, so the loopback gate rejects it. The container's only
  documented setup path (printed in its own startup banner) is
  unreachable in the most common containerized deployment.
- **Workaround that actually works:** `podman exec` into the
  container and curl `127.0.0.1:8888` from there — but no user
  reading the banner would think to do this. The banner suggests
  SSH tunneling, which is irrelevant when termigate is on the
  same host.
- **Suggested fixes (any one would work):**
  1. Drop the loopback gate when a valid setup token is present
     (the unguessable token is already the auth factor).
  2. Make the loopback check permissive for RFC1918 source IPs
     when the env hints we're in a container.
  3. Document a `podman exec` recipe in the startup banner /
     deploy docs; do not advertise the setup URL as the path.
- **Drive impact:** I worked around this by `podman exec`-ing into
  the container and submitting the form from inside, so the rest
  of the drive could proceed. A real user installing the
  containerised release would be stuck.

### F-03 — Container's `termigate` user has `/usr/sbin/nologin` as login shell, so every tmux pane dies on creation

- **Severity:** blocker
- **Repro:**
  1. Run the container (`make run-container`).
  2. Log in as admin, click `New Session`, give a valid name,
     submit, then attach.
- **Expected:** The terminal opens with a live shell prompt.
- **Actual:** The session row appears with `1 window · 0 panes`,
  the attach view shows `No panes in this window`, and the
  Quick Action pills are all disabled.
- **Root cause:** `Containerfile:30` creates the user with
  `--shell /usr/sbin/nologin`. Termigate runs `tmux new-session
  -d -s name -x C -y R` with no `SHELL` env override, so tmux
  invokes the user's login shell from `/etc/passwd`. `nologin`
  exits immediately with `This account is currently not
  available`, so the pane is dead the moment it spawns. Once
  the only pane in the window dies, tmux removes the window,
  removes the session, and the daemon shuts down (the leftover
  socket inode in `/tmp/tmux-10001/default` is a tell).
- **Verified workaround:** `SHELL=/bin/sh tmux new-session -d
  -s probe -x 80 -y 24 sh` from inside the container keeps the
  pane alive and `tmux capture-pane` returns expected output.
  The container ships both `/bin/sh` and `/bin/bash`.
- **Fix options (any one):**
  1. `Containerfile:30`: change `--shell /usr/sbin/nologin` to
     `--shell /bin/bash` (or `/bin/sh`). The container is already
     intended to host interactive shells; nologin contradicts the
     entire point of the image.
  2. `deploy/container-entrypoint.sh`: `export SHELL=/bin/bash`
     before launching the BEAM. Cheaper / more localised.
  3. `Termigate.Tmux.*`: pass `SHELL` into the env when invoking
     `tmux new-session`. Most defensive but biggest change.
- **Drive impact:** Step 2d (streaming pipeline) is unrunnable as
  shipped. To verify the rest of the application logic still
  works, I restarted the container with `SHELL=/bin/bash` injected
  via `-e` and continued. Findings after this point assume the
  workaround.

### F-04 — `make run-container` mounts the wrong path; admin/config never persists

- **Severity:** blocker
- **Repro:**
  1. `make run-container` (the documented happy path).
  2. Create an admin via `/setup`. Stop the container. Start it
     again with the same command.
- **Expected:** Admin credentials (and any YAML edits the user
  made) survive the restart. The container is the deploy unit;
  losing config on `--rm` is unacceptable.
- **Actual:** On the second run, `Config file not found, creating
  defaults at /var/lib/termigate/config.yaml` is logged again,
  there is no admin, and the user must re-run setup.
- **Root cause:** Two mismatched paths.
  - `Containerfile`: `ENV TERMIGATE_CONFIG_PATH=/var/lib/termigate/config.yaml`
    and `VOLUME ["/var/lib/termigate"]`.
  - `Makefile:run-container`: `-v "${CONFIG_DIR}":/home/termigate/.config/termigate:Z`.
  The Makefile mounts the host `${CONFIG_DIR}` over an
  unrelated path that the running app never reads. The actual
  config file lives in the image-internal anonymous-volume layer
  and dies with `--rm`.
- **Verified during this drive:** `ls /var/lib/termigate/` inside
  the running container shows `config.yaml` (5562 bytes); `ls
  /home/termigate/.config/termigate/` (the mount target) is
  empty.
- **Suggested fixes (any one):**
  1. Change the Makefile mount to `-v "${CONFIG_DIR}":/var/lib/termigate:Z`.
  2. Change the Containerfile env to
     `ENV TERMIGATE_CONFIG_PATH=/home/termigate/.config/termigate/config.yaml`
     and adjust `VOLUME` to match. (Less surprising for users
     who already have host configs in `~/.config/termigate`.)
- **Note:** `cbad11a` (password hashing migration) and `095573f`
  (auth_version session binding) lose any rollback story under
  this bug — the credentials they hash are thrown away on every
  container restart.

### F-05 — Browser terminal WebSocket is rejected with 403; xterm.js never receives data

- **Severity:** blocker
- **Repro:**
  1. Log in as admin, attach to any session.
  2. Observe the terminal area is blank, even though
     `tmux capture-pane -p` from inside the container shows the
     prompt is alive.
  3. DevTools console shows a stream of `WebSocket connection to
     'ws://127.0.0.1:8889/socket/websocket?vsn=2.0.0' failed:
     Error during WebSocket handshake: Unexpected response code:
     403`. The server log shows `WebSocket auth failed from
     192.168.2.2 / REFUSED CONNECTION TO TermigateWeb.UserSocket`
     for every retry.
- **Expected:** WebSocket connects, terminal channel joins,
  bidirectional input/output streams.
- **Actual:** Every WS upgrade is rejected. Keystrokes typed in
  the browser never reach tmux (verified by `tmux capture-pane`
  before and after typing). Initial pane state is never replayed
  to the browser.
- **Diagnosis:** The recent change `c21f1a0` ("Move browser
  channel auth off URL query tokens") moved browser auth from
  URL token to the signed Plug session cookie via
  `connect_info: [session: @session_options]`. The unit tests in
  `server/test/termigate_web/channels/user_socket_test.exs` only
  exercise the path where the session map is *already decoded*
  (passed in directly as `connect_info: %{session: session}`).
  They do not exercise the production path where Plug.Session
  must decrypt the real `Cookie:` header during the WS upgrade.
  - I confirmed end-to-end the cookie is correct: a Python script
    that POSTs to `/login`, captures the `_termigate_key` cookie,
    decodes it via `/app/bin/termigate rpc Plug.Session.COOKIE.get/3`
    inside the container, recovers `auth_version =
    "CeLtLzsOaYZTdcVO"` and `authenticated_at = 1777809175`. Both
    match what `Termigate.Auth.auth_version()` returns *now* and
    are well within `session_ttl_seconds`. So
    `cookie_authenticated?` would return `true` if the session
    map were available in `connect_info`.
  - But sending that exact cookie on a raw WS upgrade still
    yields `403` with `WebSocket auth failed`. This means
    `connect_info[:session]` is empty (or unparsed) at WS
    upgrade time on this Phoenix 1.8.5 + Bandit 1.10.3 build.
- **Surface area lost:** Step 2d (streaming pipeline), Step 2e
  (multi-tab mirroring, Quick Action pills which depend on the
  channel), Step 2f's `/multi-pane` LiveView channel — all
  unrunnable until this is fixed.
- **Suggested fix:** Add a real integration test that performs a
  WebSocket upgrade against the live endpoint with a Cookie
  header (Bandit + Plug.Session decode together), and either
  fix the session-decoding wiring on `/socket` or pin the
  cookie decode in connect/3 explicitly (e.g. fetch the
  `cookie` header from `:x_headers` and call
  `Plug.Session.COOKIE.get/3` inline).

### F-06 — Kill Session confirm modal never opens; tmux session is never killed

- **Severity:** major
- **Repro:**
  1. Log in, open the session list, click the kebab on any
     session row, click `Kill Session`.
- **Expected:** A confirmation modal appears; on confirm, the
  session is killed and the row disappears from the list.
- **Actual:** Nothing visible happens. The session stays in
  the list (`tmux ls` inside the container confirms it is
  still alive). No further click on `Kill Session` does
  anything either.
- **Root cause:** `session_list_live.ex:103` handles
  `request_kill_session` by assigning
  `:confirm_action = :kill_session`, which re-renders a
  `<dialog id="confirm-modal" class="modal">` element. The
  DOM contains the dialog, but `dialog.open === false` and
  the `open` attribute is missing, so nothing is visually
  surfaced and the user cannot confirm. Kill Session is
  effectively dead code in the browser.
- **JS check:** verified via `evaluate_script` —
  `document.getElementById('confirm-modal')` exists,
  `innerHTML.length === 13507`, but `.open === false`. The
  `request_kill_session` handler doesn't push a JS command
  (`Phoenix.LiveView.JS.dispatch/2` to call `showModal()`,
  or similar) and there is no `phx-mounted` hook on the
  dialog node.
- **Fix options:**
  1. Wrap the dialog with a `phx-mounted={JS.dispatch("showModal", ...)}`
     trigger keyed off `@confirm_action`.
  2. Add a JS hook that observes the assigns and calls
     `el.showModal()` / `el.close()` when state changes.
  3. Replace `<dialog>` with a `<div class="modal modal-open">`
     toggle (DaisyUI's CSS-only path).
- **Drive impact:** Step 2e session lifecycle cannot be
  exercised through the UI; I had to fall back to
  `podman exec termigate tmux kill-session -t drive-test`.

### F-07 — Settings page tells the user the wrong config path

- **Severity:** minor
- **Repro:** Visit `/settings`, scroll to `CONFIG FILE`.
- **Expected:** The path shown matches where the user can edit
  the file from outside the container.
- **Actual:** UI says *"Stored at `/var/lib/termigate/config.yaml`"*.
  Inside the container that is the real path, but a user running
  the official `make run-container` will not find that path on
  their host (and the path on the host the Makefile *does* mount,
  `${CONFIG_DIR}` → `~/.config/termigate`, isn't shown). Combined
  with F-04 this is the difference between "I can't find my
  config" and "my config doesn't actually persist."
- **Fix:** Either show the host-side mount path (requires the
  app to know where the volume came from) or extend the
  description with a hint like *"inside the container"* and
  document the host-side path in the deploy README.

### Top 3 user-impacting issues

1. **F-05 — Terminal WebSocket is rejected with 403.** The core
   feature of the product (browser-based terminal streaming) does
   not function in any browser session on the current `main`
   build. xterm.js sees nothing; keystrokes never reach tmux.
2. **F-03 — Container's `termigate` user has `/usr/sbin/nologin`
   shell, so every pane dies the instant it spawns.** Even
   without F-05 present, no real shell would ever run inside a
   newly-created session.
3. **F-04 — `make run-container` mounts the wrong path.**
   Admin credentials and YAML configuration vanish on every
   container restart, defeating any persistence story for the
   official packaged release.

### Ready to ship: **no**

Three independent container-side blockers (F-03, F-04, F-05)
each on their own would prevent the documented happy path from
working. Together they make the production container release
unusable as packaged.
