## Findings

### F1 — Quick Action Bar has no fade-edge / scroll affordance on the web

- **Severity:** minor
- **Repro:**
  1. Open a session terminal page in a viewport narrow enough that the
     Quick Action pills overflow horizontally (e.g. ~390 px wide, or
     when the user installs more pills).
  2. Inspect the `.quick-action-bar` element.
- **Expected:** A fade-edge (mask-image / linear-gradient) on
  whichever side has more pills off-screen, mirroring the recent
  Android fix in commit `1c9e29b` (Bug 6 in `ANDROID_DRIVE_01.md`).
- **Actual:** `mask-image: none`, no `::before` / `::after` overlays.
  The bar uses plain `overflow-x: auto`, so on narrow viewports the
  pills hard-clip at the viewport edge with no visual cue that more
  pills exist. Confirmed via `getComputedStyle` and DOM inspection
  (`scrollWidth` 568 > `clientWidth` 380, no mask). The `1c9e29b`
  fix was applied only to `android/.../QuickActionBar.kt`; the web
  side (`server/lib/termigate_web/live/multi_pane_live.ex` +
  `server/assets/css/app.css:445 .quick-action-bar { … }`) was not
  touched.
- **Note:** This is a parity gap, not a regression — the web has
  always been like this — but worth fixing for the same UX reason
  the Android fix was made.

### F2 — Split panes after the first lack a visible / accessible label

- **Severity:** minor (a11y + multi-pane usability)
- **Repro:**
  1. Open a session, click "Split horizontally" (or vertically).
  2. Two terminals render side-by-side.
  3. Query the page for `pane N` labels.
- **Expected:** Each pane has its own visible label (e.g.
  `pane 0` / `pane 1`) so the user — and screen readers — can tell
  which pane has focus.
- **Actual:** Only `pane 0` has a label. The second pane has none.
  `document.querySelectorAll('.xterm-helper-textarea').length === 2`
  but only one element matches `/^pane \d+$/`. Screenshot:
  `drive-artifacts/06_split_horizontal.png`.

### F3 — Confirm modal element stays in the a11y tree when hidden (nit)

- **Severity:** nit
- **Repro:**
  1. From session list, expand a session's "Session actions",
     click "Kill Session", then "Confirm".
  2. Run a `take_snapshot` afterward.
- **Expected:** The closed dialog should not appear in the
  accessibility tree (or at least not as `dialog modal`).
- **Actual:** The dialog is correctly hidden visually
  (`visibility: hidden`, `aria-hidden=true`), but the snapshot
  tool still surfaces it as `dialog modal`, which is at minimum
  a screen-reader smell. Likely just an `aria-hidden` not being
  enough — the `role="dialog"` on a `display: grid` element is
  still discoverable. Consider `display: none` or removing the
  node when closed (the recent server-side-modals refactor in
  `49d2e81` keeps the element mounted).

### F4 — Default Chromium MCP `localhost` resolves to ::1, which podman doesn't bind

- **Severity:** nit (environmental; not a termigate bug)
- **Note:** Recording for next driver. Rootless podman binds the
  port-forward to IPv4 only, but `localhost` on this host resolves
  to `::1` first, so `curl http://localhost:8889/healthz` and any
  browser nav using `localhost` fails with "Connection reset by

  `127.0.0.1:8889` throughout.

**Top 3 user-impacting issues**

1. **F1** — On narrow viewports / extra pills, the Quick Action bar
   hard-clips with no fade affordance, so users do not know more
   pills exist. Same UX bug the Android side fixed in `1c9e29b`.
2. **F2** — Split panes after the first lack a `pane N` label,
   making it harder to orient (and worse for screen readers) once
   the user splits.
3. **F3** — The closed Confirm modal is hidden visually but still
   shows up as `dialog modal` in the accessibility tree.

**Verdict:** `yes-with-caveats` — the core pipeline (auth, session
lifecycle, streaming, multi-tab mirror, multi-pane, settings) is
solid. The two minor findings are paper cuts, not blockers.
